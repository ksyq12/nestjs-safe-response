## Findings

### 1. Nullable pagination strings are documented as `object` in OpenAPI

Severity: High

Files:

- `src/dto/response.dto.ts`
- `test/e2e/__snapshots__/swagger-schema.e2e-spec.ts.snap`

Evidence:

- `PaginationLinksDto.prev`, `next`, `last` are declared as `string | null` without an explicit Swagger scalar type.
- `CursorPaginationMetaDto.nextCursor` and `previousCursor` are declared the same way.
- The generated OpenAPI snapshot records those fields as `type: "object"` instead of `type: "string", nullable: true`.

Impact:

- Generated SDKs and schema consumers will infer the wrong type for common pagination metadata.
- The runtime contract is correct, but the published Swagger contract is not.

Recommended fix:

- Add explicit Swagger scalar typing to nullable string fields, for example `type: String` with `nullable: true`.
- Add a regression assertion in Swagger unit/E2E coverage so these fields stay `string | null`.

### 2. Error/Problem `meta` schema does not explicitly describe arbitrary CLS fields

Severity: Medium

Files:

- `src/dto/response.dto.ts`
- `src/interfaces/index.ts`
- `README.md`
- `test/e2e/__snapshots__/swagger-schema.e2e-spec.ts.snap`

Evidence:

- Runtime/types allow arbitrary CLS-derived metadata fields such as `traceId` and `correlationId`.
- `ErrorResponseMetaDto` uses an index signature, but the generated Swagger schema still shows only fixed properties (`responseTime`, `deprecation`, `rateLimit`).
- The docs promise arbitrary context metadata in `meta`.

Impact:

- Schema readers and generated clients will not see those context fields as part of the documented contract.
- This is primarily a documentation/codegen issue; runtime behavior is unaffected.

Recommended fix:

- Make the OpenAPI schema explicitly permissive for additional `meta` keys, and add a regression check against the emitted schema.

### 3. README test inventory counts are stale

Severity: Low

Files:

- `README.md`
- `README.ko.md`

Evidence:

- Both READMEs still advertise `388` unit tests, `44/44/40` E2E splits, and `84` type tests.
- Current verified totals are `427` unit tests and `143` E2E tests total.

Impact:

- Public release docs no longer match the actual verification state.
- This does not affect runtime stability.

Recommended fix:

- Update the counts or switch the table wording to approximate/maintained values that do not drift every release.
