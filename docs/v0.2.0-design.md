# v0.2.0 설계 문서 — 멀티 컨텍스트 지원

## 1. 현재 문제점

### 1.1 Express 하드코딩

Interceptor와 Filter 모두 Express 타입을 직접 import하고 있음:

```typescript
// safe-response.interceptor.ts (line 11)
import { Request } from 'express';

// safe-exception.filter.ts (line 10)
import { Response, Request } from 'express';
```

Fastify를 사용하는 NestJS 프로젝트에서는 `request.url`과 `response.status().json()` 인터페이스가 다름.

### 1.2 HTTP 컨텍스트 전용

```typescript
// interceptor (line 55-56)
const request = context.switchToHttp().getRequest<Request>();
const statusCode = context.switchToHttp().getResponse().statusCode;

// filter (line 25-27)
const ctx = host.switchToHttp();
```

WebSocket이나 RPC 컨텍스트에서 `switchToHttp()`를 호출하면 런타임 에러 발생.

## 2. 해결 방안

### 2.1 Express 의존 제거 — 플랫폼 무관 타입 사용

Express/Fastify 모두 NestJS의 HTTP adapter를 통해 동일한 인터페이스를 제공.
`context.switchToHttp().getRequest()`는 어댑터에 관계없이 `url` 프로퍼티를 가지고 있음.
`context.switchToHttp().getResponse()`는 어댑터에 관계없이 `statusCode`를 가짐.

**변경**: Express 타입 import를 제거하고, 필요한 프로퍼티만 추출하는 방식으로 전환.

```typescript
// Before
import { Request } from 'express';
const request = context.switchToHttp().getRequest<Request>();

// After (플랫폼 무관)
const request = context.switchToHttp().getRequest();
const url: string = request.url;
```

Filter의 `response.status(code).json(body)` 체이닝은:
- Express: `res.status(code).json(body)` → 동작
- Fastify: `reply.status(code).send(body)` → `.json()`이 없음

**해결**: NestJS의 `HttpAdapterHost`를 사용하거나, `response.status().send()`로 통일하지 않고,
실제로는 Fastify의 reply 객체도 `.json()` 없이 `.send()`로 JSON을 보낼 수 있음.
가장 안전한 방법: `response` 객체의 메서드 존재 여부로 분기.

```typescript
// 플랫폼 무관 응답
const res = host.switchToHttp().getResponse();
if (typeof res.status === 'function') {
  // Express: res.status(code).json(body)
  res.status(statusCode).json(body);
} else if (typeof res.code === 'function') {
  // Fastify: reply.code(code).send(body)
  res.code(statusCode).send(body);
}
```

더 깔끔한 대안: `HttpAdapterHost`를 주입받아 `httpAdapter.reply(res, body, statusCode)` 사용.
이 방식이 NestJS 공식 권장 패턴.

```typescript
constructor(
  private readonly httpAdapterHost: HttpAdapterHost,
) {}

catch(exception, host) {
  const { httpAdapter } = this.httpAdapterHost;
  const ctx = host.switchToHttp();
  httpAdapter.reply(ctx.getResponse(), body, statusCode);
}
```

### 2.2 ExecutionContext 분기 — HTTP 이외 컨텍스트 보호

```typescript
intercept(context: ExecutionContext, next: CallHandler) {
  const contextType = context.getType(); // 'http' | 'rpc' | 'ws'

  if (contextType !== 'http') {
    return next.handle(); // HTTP가 아니면 래핑하지 않고 통과
  }

  // ... 기존 HTTP 래핑 로직
}
```

Filter도 동일:
```typescript
catch(exception, host) {
  const contextType = host.getType();

  if (contextType !== 'http') {
    throw exception; // HTTP가 아니면 원래 에러를 그대로 전파
  }

  // ... 기존 HTTP 에러 처리 로직
}
```

## 4. 추가 테스트 케이스

### Interceptor
- contextType이 'rpc'이면 래핑 없이 원본 반환
- contextType이 'ws'이면 래핑 없이 원본 반환
- contextType이 'http'이면 정상 래핑 (기존 동작)

### Filter
- contextType이 'rpc'이면 예외를 그대로 throw
- contextType이 'ws'이면 예외를 그대로 throw
- contextType이 'http'이면 정상 에러 응답 (기존 동작)
- HttpAdapterHost를 통한 reply 호출 검증

### E2E (Fastify)
- Fastify 어댑터로 NestJS 앱 부팅 → GET /test → 래핑된 응답 확인
- Fastify 어댑터 → GET /test/not-found → 에러 응답 확인
