# v0.4.0 설계 문서 — 안정성

**현재 버전: 0.3.0 → 목표 버전: 0.4.0**

## 개요

nestjs-safe-response v0.4.0은 소스 코드 변경 없이 호환성 검증과 엣지 케이스 테스트를 추가한다:

1. **NestJS v10/v11 CI 매트릭스** — peerDeps에 선언된 v10 호환성을 실제 CI에서 검증
2. **@nestjs/swagger v7/v8/v11 CI 매트릭스** — 데코레이터/DTO가 swagger 버전별로 정상 동작하는지 검증
3. **엣지 케이스 테스트** — 비정형 데이터(undefined, Buffer, 빈 값 등) 반환 시 동작 보장

소스 코드를 변경하지 않으며, CI와 테스트만 확장한다. v1.0.0 API 안정화 선언의 근거가 된다.

---

## 1. CI 매트릭스 확장

### 현재 상태

`ci.yml`은 Node.js 버전(18, 20, 22)만 매트릭스로 실행한다. NestJS와 swagger는 `package.json`에 고정된 단일 버전(NestJS 11, swagger 11)으로만 테스트된다.

### 변경 후

```yaml
strategy:
  matrix:
    node-version: [18, 20, 22]
    nestjs-version: [10, 11]
    swagger-version: [7, 8, 11]
    exclude:
      # NestJS v10은 swagger v11과 호환 안 됨 (swagger v11은 NestJS v11 전용)
      - nestjs-version: 10
        swagger-version: 11
```

### 매트릭스 조합 (총 15개)

각 행은 3개의 Node.js 버전(18, 20, 22)을 포함하므로 행당 3개 조합이다.

| Node | NestJS | Swagger | 예상 | 조합 수 |
|------|--------|---------|------|---------|
| 18/20/22 | 11 | 11 | 통과 (현재 테스트 환경) | 3 |
| 18/20/22 | 11 | 8 | 검증 필요 | 3 |
| 18/20/22 | 11 | 7 | 검증 필요 | 3 |
| 18/20/22 | 10 | 8 | 검증 필요 | 3 |
| 18/20/22 | 10 | 7 | 검증 필요 | 3 |

### 버전 덮어쓰기 메커니즘

기본 `npm ci` 후 매트릭스 값에 따라 해당 버전을 덮어쓰기 설치한다.

**기본 버전(nestjs-version: 11, swagger-version: 11)은 `npm ci` 결과를 그대로 사용한다.** Override 스텝은 기본이 아닌 버전에서만 실행된다.

```yaml
- name: Override NestJS version
  if: matrix.nestjs-version != 11
  run: |
    npm install --no-save \
      @nestjs/common@^${{ matrix.nestjs-version }} \
      @nestjs/core@^${{ matrix.nestjs-version }} \
      @nestjs/platform-express@^${{ matrix.nestjs-version }} \
      @nestjs/testing@^${{ matrix.nestjs-version }}

- name: Override Swagger version
  if: matrix.swagger-version != 11
  run: |
    npm install --no-save @nestjs/swagger@^${{ matrix.swagger-version }}
```

`--no-save`를 사용하여 `package.json`을 변경하지 않는다.

### 실패 시 대응 방침

- 특정 조합에서 실패하면 원인을 분석하여:
  - API 차이가 작으면 호환 코드 추가
  - API 차이가 크면 peerDeps 범위를 좁힘 (예: `^10` 제거)
  - swagger 특정 버전에서만 실패하면 해당 버전 peerDeps 제거

## 2. 엣지 케이스 테스트

### 유닛 테스트 (interceptor.spec.ts에 추가)

**참고**: `null` 반환 케이스는 기존 유닛 테스트(`interceptor.spec.ts` 117-127행)에서 이미 커버되므로 중복 추가하지 않는다.

| 테스트 케이스 | 입력 | 기대 결과 |
|--------------|------|-----------|
| `undefined` 반환 | `undefined` | `data` 키가 없거나 undefined (JSON 직렬화 시 `undefined` 값은 필드 자체가 생략됨) |
| 빈 객체 반환 | `{}` | `data: {}`로 래핑 |
| 빈 배열 반환 | `[]` | `data: []`로 래핑 |
| 숫자 반환 | `42` | `data: 42`로 래핑 |
| boolean 반환 | `true` | `data: true`로 래핑 |
| 중첩 객체 반환 | `{ a: { b: { c: 1 } } }` | `data`에 중첩 구조 그대로 유지 |

### E2E 테스트 (safe-response.e2e-spec.ts에 추가)

| 테스트 케이스 | 라우트 | 기대 결과 |
|--------------|--------|-----------|
| `undefined` 반환 | `GET /test/edge-undefined` | `res.body`에 `data` 프로퍼티가 없음 (`JSON.stringify`가 `undefined` 값을 생략) |
| `null` 반환 | `GET /test/edge-null` | `data: null` |
| Buffer 반환 | `GET /test/edge-buffer` | Buffer가 JSON 직렬화되어 `data`에 `{ type: 'Buffer', data: [...] }` 형태로 포함됨. 이는 바이너리 데이터가 손상되는 예시로, README에서 `@RawResponse()` 사용을 안내하는 근거 |
| 빈 문자열 반환 | `GET /test/edge-empty-string` | `data: ""` |

### E2E 테스트 인프라 추가

`test/e2e/test.controller.ts`에 4개 라우트 추가:

```typescript
@Get('edge-undefined')
edgeUndefined() {
  return undefined;
}

@Get('edge-null')
edgeNull() {
  return null;
}

@Get('edge-buffer')
edgeBuffer() {
  return Buffer.from('hello');
}

@Get('edge-empty-string')
edgeEmptyString() {
  return '';
}
```

### Buffer/Stream 처리 방침

Buffer나 Stream을 반환하는 경우 `@RawResponse()`를 사용해야 한다. 인터셉터가 이들을 래핑하면 JSON 직렬화에서 바이너리 데이터가 `{ type: 'Buffer', data: [104, 101, ...] }` 형태로 변환된다. 자동 감지보다 명시적 `@RawResponse()` 사용을 권장하며, 이를 README에 안내한다.
